Why use this engine

Used appropriately, the engine helps researchers produce more comparable depression measures from SSQ-10 data while keeping the scoring workflow reproducible and transparent.

  • It places SSQ-10 responses onto a harmonised depression scale rather than relying only on a raw total score.
  • It returns both continuous and binary outputs, which supports different reporting needs across studies.
  • It makes the applied threshold explicit, including subgroup-specific targets where those are available.
  • It supports batch scoring with the same packaged logic, which improves consistency across participants and cohorts.
  • It helps researchers reuse existing SSQ-10 data without rerunning the full psychometric workflow each time.

Important limitations

This tool is intended to support harmonised research scoring. It is not a clinical diagnosis tool, and its outputs should be interpreted alongside study context, local governance, and appropriate subject-matter oversight.

  • The app scores binary SSQ-10 responses only. It does not accept SSQ-14 or PHQ-9 responses as direct input.
  • Current production-safe engines share the same 1-factor SSQ-10 scorer, so changing engines often changes the applied threshold rather than the harmonised theta itself.
  • Packaged performance metrics come from the packaging sample and may not generalise unchanged to other cohorts, settings, languages, age ranges, or treatment-exposed groups.
  • If subgroup fields needed for a subgroup-specific threshold are missing or unavailable, the app falls back to the stored overall threshold and records the applied target in the output.
  • This browser build has been smoke-tested against the R engine, but it remains a browser implementation and should be version-controlled alongside its validation summary.

Privacy

The intended browser deployment pattern is privacy-first: raw uploads stay within the active session and only user-requested exports are returned.

What the app does

  • Loads the selected engine and validates the uploaded schema.
  • Scores the data in memory during the active browser session.
  • Returns harmonised outputs directly to the user.
  • Generates downloadable files only when the user requests them.

What the app does not do

  • It does not write raw uploaded datasets back into the project repository.
  • It does not persist session inputs to GitHub or project storage.
  • It should not log raw item-level responses in production telemetry.
  • It is not designed to replace governance for regulated or restricted data environments.

Documentation

Use this view to inspect the expected input contract and the interpretation of the returned harmonised fields.

Scoring outputs

  • SSQ10_Total_Score is the observed raw sum of endorsed SSQ-10 items.
  • Theta_Harmonized is the underlying harmonised depression score.
  • Depression_Binary applies the stored engine-specific cutoff strategy.
  • PHQ_Expected_Sum maps the harmonised theta back into a PHQ-oriented scale.
  • PHQ_Prob_GE10 estimates the probability of being at or above the PHQ-9 >= 10 threshold.
  • The returned output also includes engine and cutoff metadata so result changes remain auditable.

Benefits and intended strengths

  • The browser version keeps scoring local to the user’s machine, which supports privacy-sensitive workflows.

Glossary: statistical terms

TermDefinition
HarmonisationThe process of making scores from different questionnaires comparable on a shared scale.
Harmonized thetaThe underlying depression score estimated by the harmonised IRT model.
SensitivityThe proportion of truly depressed participants correctly identified by the engine.
SpecificityThe proportion of truly non-depressed participants correctly identified by the engine.
AccuracyThe overall proportion of classifications that are correct.
Depression prevalenceThe proportion of scored participants classified as depressed in the current scored dataset.
Cutoff thresholdThe theta value above which the engine classifies a participant as depressed.
Threshold approachThe rule used to choose the decision threshold, for example prioritising sensitivity or specificity.
Threshold scopeThe grouping level at which thresholds are applied, such as overall, by sex, by age group, or by sex plus age group.
Cohen's kappaA measure of agreement between the engine classification and the reference classification after accounting for agreement expected by chance.
